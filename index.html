1.Понятие данного.

Данное: элемент информации, содержащий значение характеристики объекта, процесса, события или явления реального мира.

Имя данного: название характеристики (например, «Фамилия»).

Значение данного: конкретное значение характеристики (например, «Иванов»).

Пример: «Фамилия студента: Иванов».

Данные связаны с объектом реального мира (например, «студент»).

 

2. Определение базы данных (БД)

Определения БД:

Совокупность данных, организованных по схеме, поддерживающая одну или более области применения.

БД: совокупность поименованных, упорядоченных и взаимосвязанных данных, организованных по правилам для хранения и манипулирования, независимо от программ.

Особенности БД:

Упорядочение исключает дублирование.

Возможность хранения данных для разных объектов (например, города).

Наличие связей между данными (в отличие от файлов).

Набор перманентных данных для прикладных систем предприятия.

Совместно используемый набор логически связанных данных для информационных потребностей организации.

 

3. Классификация баз данных

Базы данных (БД) классифицируются по различным критериям, включая модель данных, среду хранения, содержимое и степень распределённости.

По модели данных

Иерархическая: данные организованы в виде дерева.

Объектная и объектно-ориентированная: данные представлены как объекты с атрибутами и методами.

Объектно-реляционная: комбинация объектной и реляционной моделей.

Реляционная: данные организованы в таблицы, связанные ключами.

Сетевая: данные связаны через графы.

Функциональная: данные обрабатываются с использованием функциональных зависимостей.

По среде постоянного хранения

Традиционная (conventional database): данные хранятся на жёстком диске (вторичная память), в оперативной памяти — только кеш и данные для обработки.

В оперативной памяти (in-memory database): все данные хранятся в оперативной памяти для быстрого доступа.

В третичной памяти (tertiary database): данные на магнитных лентах или оптических дисках, с каталогом во вторичной памяти сервера.

 

По содержимому

Географическая

Историческая

Научная

Мультимедийная

Клиентская

 

По степени распределённости

Централизованная: БД на одном компьютере.

Распределённая: данные размещены на разных узлах сети.

Однородная: все узлы используют одну СУБД.

Неоднородная: разные узлы используют разные СУБД.

 

Отсылочная БД: содержит ссылки на другие источники.

Адресно-справочная: указывает адреса хранения данных.

Документальная: записи содержат библиографическое описание документа.

Библиографическая: отсылочная БД с библиографическими записями.

Реферативная: библиографическая БД с аннотациями или рефератами.

Фактографическая (база первичных данных): содержит данные о предметной области.

Объектографическая: данные об отдельных объектах предметной области.

Полнотекстовая: содержит полные тексты документов.

База данных показателей: данные о свойствах материалов, процессов, явлений.

Лексикографическая: записи соответствуют статьям словаря.

Численная: содержит числовые данные.

Численно-текстовая: комбинация числовых и текстовых данных.

Текстовая: преимущественно текст на естественном языке.

Гипертекстовая: текстовая БД с логическими связями между записями.

 

Другие виды БД

Пространственная: поддерживает пространственные свойства (используется в ГИС).

Временная (темпоральная): учитывает аспекты времени.

Пространственно-временная: комбинирует пространство и время.

Циклическая: фиксированный объём, старые данные заменяются новыми циклически.

 

4.История возникновения и развития баз данных.

История возникновения БД

Две области применения вычислительной техники:

Численные расчеты: сложные алгоритмы, простые данные, малый объем.

Информационные системы: большие объемы данных, сложная структура, функции:

Надежное хранение.

Специфические преобразования и вычисления.

Удобный интерфейс.

 

Файловые системы:

Файл — именованная область памяти с линейной последовательностью записей.

Операции: создание, открытие, чтение, запись.

Недостатки:

Зависимость программ от структуры данных (изменение структуры файла требует перекомпиляции программ).

Отсутствие централизованного управления доступом.

Проблемы с параллельной работой пользователей (блокировка при модификации).

 

Необходимость СУБД: устранение зависимости программ от данных, управление доступом, поддержка многопользовательского режима.

 

История развития БД

Первый этап (начало 60-х):

Хранение данных на магнитных лентах в последовательных файлах.

Физическая структура соответствует логической.

Файлы для одной программы, изменения требуют переработки программ.

 

Второй этап (середина 60-х):

Появление устройств прямого доступа (произвольный доступ к записям).

Поиск по ключевому полю.

Перенос файлов без изменения программ.

 

Третий этап (конец 60-х):

Централизация данных для разных приложений.

Уменьшение избыточности и противоречивости.

Появление иерархических и сетевых СУБД (данные в виде деревьев и графов).

Эффективный поиск по многим ключам.

 

Четвертый этап (вторая половина 70-х):

Логическая и физическая независимость данных.

Безопасность, целостность, удобство развития БД.

Поиск по различным запросам.

Языковые средства для администраторов, программистов, пользователей.

Начало исследований реляционной модели (Э. Кодд, таблицы).

 

Этап персональных компьютеров (80-е):

Дружественный интерфейс (меню, окна, контекстная помощь).

Упрощение схем СУБД.

Переход на реляционные СУБД.

Ориентация на непрофессиональных пользователей.

Средства автоматизации (генераторы форм, отчетов, запросов).

5. Сверхбольшие базы данных (VLDB)

Сверхбольшая база данных — это база данных, которая занимает чрезвычайно большой объём на устройстве физического хранения. Термин подразумевает максимально возможные объёмы БД, которые определяются последними достижениями в технологиях физического хранения данных и в технологиях программного оперирования данными.

Эволюция объёмов:

1997: 7 ТБ (Knight Ridder's DIALOG).

2001: 10,5 ТБ.

2003: 25 ТБ.

2005: ~100 ТБ.

2006: 850 ТБ (Google).

2010: измеряется в петабайтах.

2014: 10–15 ЭБ (Google).

2025 (прогноз): 2–40 ЭБ для геномных данных.

 

Особенности:

Требуются специальные решения для дисковых подсистем, операционных сред и СУБД.

Исследования поддерживаются конференцией VLDB и организацией VLDB Endowment.

 

Единицы измерения (ГОСТ 8.417–2002):

Байт (Б): 10⁰

Килобайт (кБ): 10³; Мегабайт (МБ): 10⁶; Гигабайт (ГБ): 10⁹; Терабайт (ТБ): 10¹²; Петабайт (ПБ): 10¹⁵; Эксабайт (ЭБ): 10¹⁸; Зеттабайт (ЗБ): 10²¹; Иоттабайт (ИБ): 10²⁴

 

 

6. Понятие хранилища данных

Хранилище данных (Data Warehouse, DW): предметно-ориентированное, неизменяемое, хронологически структурированное собрание данных для поддержки управленческих решений (по У. Инмону).

Многомерное представление данных:

Реляционная модель: данные в двумерных таблицах, связанных ключами.

Пример: таблица с моделями автомобилей, месяцами и объёмами продаж (360 строк для 30 моделей и 12 месяцев).

Многомерная модель: данные в виде гиперкуба (например, 12×30 таблица для тех же данных).

OLAP-куб: трёхмерная модель (например, измерения: Время, Товар, Регион; показатель: Объём продаж).

 

Основные понятия многомерной модели:

Измерение (Dimension): множество однотипных данных (например, Годы, Регионы).

Показатель (Measure): значение, определяемое измерениями (например, Объём продаж).

Переменная: значения вводятся или формируются и хранятся.

Формула: значения вычисляются по формуле.

 

Пространственная интерпретация: данные отображаются в виде координат в многомерном пространстве (например, трёхмерный куб).

7.              Жизненный цикл БД.

Под жизненным циклом базы данных понимаются этапы развития БД, начиная от анализа предметной области, и заканчивая эксплуатацией БД. 1.Исследование и анализ проблемы, для решения которой создаётся база данных. 2. Построение Инфологической и Даталогической модели. 3.  Нормализация полученных Инфологических и Даталогических моделей. По окончании этого этапа, как правило получают заготовки таблицы БД и набор связей между ними (первичные и вторичные ключи). 4.  Проверка целостности БД 5. Выбор физического способа хранения и эксплуатации (тех. средства) базы данных. 6.  Проектирование входных и выходных форм. 7.  Разработка интерфейса приложения. 8.  Функциональное наполнение приложения. 9.  Отладка: проверка на корректность работы функционального наполнения системы. 10.Тестирование: тест на корректность ввода вывода данных, тест на максимальное количество активных сессий и т. д. 11.Ввод в эксплуатацию: отладка ИТ-инфраструктуры, обучение пользователей и ИТ-персонала. 12.При необходимости добавления выходных форм и дополнительной функциональности. В случае если необходимы более серьёзные изменения, следует повторить все шаги с первого. 13. Вывод из эксплуатации: перенос данных в новую СУБД.

Кратко основные этапы жизненного цикла базы данных: проектирование БД, проектирование приложений, реализация БД, разработка специальных средств администрирования БД, эксплуатация.

8.              Методологические основы баз данных

Методологические основы баз данных включают в себя несколько этапов:

1.              Концептуальное (инфологическое) проектирование. Построение формализованной модели предметной области с использованием стандартных языковых средств, обычно графических, например ER-диаграмм (диаграмм «Сущность-связь»). Модель строится без ориентации на какую-либо конкретную систему управления базами данных (СУБД). Основные элементы модели: описание объектов предметной области и связей между ними, информационных потребностей пользователей, алгоритмических зависимостей между данными, ограничений целостности (требований к допустимым значениям данных и к связям между ними).

2.              Логическое (даталогическое) проектирование. Отображение инфологической модели на модель данных, используемую в конкретной СУБД, например на реляционную модель данных. Для реляционных СУБД даталогическая модель — набор таблиц, обычно с указанием ключевых полей, связей между таблицами. 3

3.              Физическое проектирование. Реализация даталогической модели средствами конкретной СУБД, а также выбор решений, связанных с физической средой хранения данных: выбор методов управления дисковой памятью, методов доступа к данным, методов сжатия данных

Процесс проектирования БД представляет собой последовательность переходов от неформального словесного описания информационной структуры предметной области к формализованному описанию объектов предметной области в терминах некоторой модели.

9.              Концептуальное проектирование.

Концептуальное проектирование - сбор, анализ и редактирование требований к данным. Для этого осуществляются следующие мероприятия:

o            обследование предметной области, изучение ее информационной структуры

o            выявление всех фрагментов, каждый из которых характеризуется пользовательским представлением, информационными объектами и связями между ними, процессами над информационными объектами

o            моделирование и интеграция всех представлений

Концептуальное (инфологическое) проектирование – построение формализованной модели предметной области. Такая модель строится с использованием стандартных языковых средств, обычно графических, например ER-диаграмм. Такая модель строится без ориентации на какую-либо конкретную СУБД.

•             Основные элементы данной модели:

•             Описание объектов предметной области и связей между ними.

•             Описание информационных потребностей пользователей (описание основных запросов к БД).

•             Описание алгоритмических зависимостей между данными.

•             Описание ограничений целостности, т.е. требований к допустимым значениям данных и к связям между ними.

По окончании данного этапа получаем концептуальную модель, инвариантную к структуре базы данных. Часто она представляется в виде модели "сущность-связь".

10.          Логическое проектирование.

Логическое проектирование - преобразование требований к данным в структуры данных. На выходе получаем СУБД-ориентированную структуру базы данных и спецификации прикладных программ. На этом этапе часто моделируют базы данных применительно к различным СУБД и проводят сравнительный анализ моделей.  Логическое (даталогическое) проектирование – отображение инфологической модели на модель данных, используемую в конкретной СУБД, например на реляционную модель данных. Для реляционных СУБД даталогическая модель – набор таблиц, обычно с указанием ключевых полей, связей между таблицами. Если инфологическая модель построена в виде ER-диаграмм (или других формализованных средств), то даталогическое проектирование представляет собой построение таблиц по определённым формализованным правилам, а также нормализацию этих таблиц. Этот этап может быть в значительной степени автоматизирован.

11.          Объектно-ориентированная модель данных.

Направление объектно-ориентированных бд (ООБД) возникло сравнительно давно. Публикации появлялись уже в середине 1980-х.

В наиболее общей и классической постановке объектно-ориентированный подход базируется на концепциях: объекта и идентификатора объекта; атрибутов и методов; классов; иерархии и наследования классов.

Любая сущность реального мира в объектно-ориентированных языках и системах моделируется в виде объекта. Любой объект при своем создании получает генерируемый системой уникальный идентификатор, который связан с объектом во все время его существования и не меняется при изменении состояния объекта.

Каждый объект имеет состояние и поведение. Состояние объекта - набор значений его атрибутов. Поведение объекта - набор методов (программный код), оперирующих над состоянием объекта. Значение атрибута объекта — это тоже некоторый объект или множество объектов. Состояние и поведение объекта инкапсулированы в объекте; взаимодействие между объектами производится на основе передачи сообщений и выполнении соответствующих методов.

Множество объектов с одним и тем же набором атрибутов и методов образует класс объектов. Объект должен принадлежать только одному классу (если не учитывать возможности наследования). Класс, объекты которого могут служить значениями атрибута объектов другого класса, называется доменом этого атрибута.

Допускается порождение нового класса на основе уже существующего класса - наследование. В этом случае новый класс, называемый подклассом существующего класса (суперкласса), наследует все атрибуты и методы суперкласса.

Идея возможного переопределения атрибутов и методов суперкласса в подклассе (перегрузки методов): эта возможность увеличивает гибкость, но порождает дополнительную проблему: при компиляции объектно-ориентированной программы могут быть неизвестны структура и программный код методов объекта, хотя его класс (суперкласс) известен. Для разрешения этой проблемы применяется так метод позднего связывания, означающий, интерпретационный режим выполнения программы с распознаванием деталей реализации объекта во время выполнения посылки сообщения к нему.

Первой формализованной и общепризнанной моделью данных была реляционная модель Кодда. В этой модели выделялись три аспекта - структурный, целостный и манипуляционный. Структуры данных в реляционной модели основываются на плоских нормализованных отношениях, ограничения целостности выражаются с помощью средств логики первого порядка и манипулирование данными осуществляется на основе реляционной алгебры или равносильного ей реляционного исчисления.

Два уровня моделирования объектов: нижний (структурный) и верхний (поведенческий). База данных — это набор элементов данных, связанных отношениями "входит в класс" или "является атрибутом". Таким образом, БД может рассматриваться как ориентированный граф.

В качестве первичных концепций схемного уровня ООБД выступают типы и классы. Отмечается, что во всех системах, использующих только одно понятие (либо тип, либо класс) это понятие неизбежно перегружено: тип предполагает наличие некоторого множества значений, определяемого структурой данных этого типа; класс также предполагает наличие множества объектов, но это множество определяется пользователем.

12.          Иерархическая модель базы данных

Иерархическая бд – самая ранняя модель представления структуры данных, эта бд представляет собой упорядоченную совокупность экземпляров данных типа «дерево», содержащих экзмепляры типа «запись». Информация здесь организована по принципу «предок-потомок». Каждая запись может иметь не более одной родительской записи и несколько подчиненных. Связи записей реализуются в виде физических указателей с одной записи на другую. Недостаток: невозможно реализоваться отношения «многие-ко-многим». Такая бд может представляться как перевернутое дерево, состоящее из объектов: верхнего уровня (корня дерева), который занимает один объект и второй уровень(объекты).

Единицей в этой модели данных являются поля. Поле данных определяется как наименьшая неделимая единица данных доступная пользователю. Для сегмента определяется тип сегмента (это именованная совокупность входящих в него типов данных) и экземпляр сегмента.

Тип «дерево» является составным. Он включает в себя подтипы (поддеревья), каждый из которых является типом «дерево». Каждый из типов «дерево» состоит из «корневого» типа и упорядоченного набора подчиненных типов. Каждый из элементарных типов, включенных в тип «дерево», является простым или составным типом «запись». Простая запись состоит из одного типа, например числового, а составная «запись» объединяет некоторую совокупность типов, например целое, строку символов и указатель.

Достоинства: эффективное использование памяти ЭВМ и неплохие показатели времени выполнения основных операций над данными. Недостатки: часто дублируется информация между записями, одной родительской записи, может соответствовать любое число дочерних.

Атрибут(элемент данных) – наименьшая единица структуры данных. Запись – наименованная совокупность атрибутов, ее использование позволяет за одно обращение к базе получить некоторую логически связанную совокупность данных. Групповое отношение – иерархическое отношение между записями двух типов. Корневая запись каждого дерева обязательно должна содержать ключ с уникальным значением.

Операции над данными, определенные в иерархической модели: добавить, удалить, извлечь.

13.          Сетевая модель базы данных.

Сетевая модель данных позволяет отображать разнообразные взаимосвязи элементов данных в виде произвольного графа, обобщая тем самым иерархическую модель данных. Связь имеет вид 1:N

Для описания схемы сетевой бд используется две группы типов: «запись» и «связь». Тип «связь» определяется для двух типов «запись»: предка и потомка. В этой модели данных запись-потомок может иметь произвольное число записей-предков.

Достоинства: возможность эффективной реализации по показателям затрат памяти и оперативности. Недостатки: высокая сложность и жесткость схемы бд.

Операции над данными в сетевой модели бд: добавить (внести запись в бд), включить в групповое отношение (связать существующую подчиненную запись с записью владельцев), переключить (связать запись), обновить (изменить значение элементов записи), извлечь (извлечь записи по значению ключа), удалить (убрать из бд), исключить из группового отношения (разорвать связь).

14. Реляционная модель базы данных

Реляционная модель данных (РМД), предложенная Э. Коддом в 1970 г., основана на математической теории множеств и использует понятие теоретико-множественного отношения. Данные представляются в виде двумерных таблиц (отношений), где:

        Строка — кортеж (запись), характеризующий конкретный объект.

        Столбец — атрибут (поле), представляющий свойство объекта.

        Домен — множество допустимых значений для атрибута (например, домен "Имена" содержит только строки, пригодные для имен).

Ключевые элементы модели:

        Схема отношения: Описание структуры таблицы в виде R(A₁, A₂, ..., Aₖ), где R — имя отношения, Aᵢ — атрибуты. Включает домены и ограничения целостности (например, "дата рождения < 200 лет").

        Первичный ключ: Набор атрибутов, однозначно идентифицирующий кортеж (минимален и неизбыточен).

        Внешний ключ: Атрибут, ссылающийся на первичный ключ другого отношения для обеспечения связей (например, Сотрудник.Отд_номер → Отдел.Отд_номер).

Фундаментальные свойства отношений:

        Атомарность значений: Значения атрибутов неделимы.

        Уникальность кортежей: Отсутствие дубликатов строк.

        Неупорядоченность: Нет заданного порядка строк/столбцов.

        Однородность столбцов: Все значения в столбце из одного домена.

Преимущества: Математическая строгость, простота восприятия, гибкость манипулирования данными через реляционную алгебру.


 

15. Виды связей БД

Связи в БД классифицируются по степени соответствия экземпляров сущностей. Основные типы:

        Один-к-одному (1:1)

            Каждому экземпляру сущности А соответствует не более одного экземпляра сущности Б, и наоборот.

            Пример: Студент ↔ Дипломная работа (один студент выполняет одну работу).

            Графическое обозначение: [Студент] ——— [Дипломная].

        Один-ко-многим (1:M) / Многие-к-одному (M:1)

            Экземпляру сущности А соответствует ноль, один или несколько экземпляров Б, но каждому Б — только один А.

            Пример: Отдел ↔ Сотрудники (в одном отделе много сотрудников).

            Графическое обозначение: [Отдел] ———< > [Сотрудник].

        Многие-ко-многим (M:N)

            Экземпляру А соответствует несколько Б, и наоборот.

            Пример: Студент ↔ Преподаватель (студент учится у многих преподавателей, преподаватель учит многих студентов).

            Реализуется через промежуточную таблицу (например, Сессия с ключами Студент_ID и Преподаватель_ID).

            Графическое обозначение: [Студент] ———< > [Преподаватель].

Важно! Тип связи влияет на проектирование таблиц (например, связь M:N требует трех таблиц).


16. Теоретико-множественные операции реляционной алгебры

Реляционная алгебра — набор операций над отношениями, результатом которых является новое отношение. Теоретико-множественные операции включают:

        Объединение (A ∪ B)

            Возвращает кортежи, принадлежащие A, B или обоим отношениям (дубликаты удаляются).

            Условие: Отношения совместимы по типу (одинаковые атрибуты и домены).

            Пример: Список всех студентов из групп 382 и 383.

        Пересечение (A ∩ B)

            Возвращает кортежи, присутствующие одновременно в A и B.

            Пример: Студенты, сдавшие и физику, и математику.

            (Лекция 7(2), стр. 2).

        Разность (A \ B)

            Возвращает кортежи из A, отсутствующие в B.

            Пример: Студенты группы 382, не сдавшие экзамен по БД.

            (Лекция 7(2), стр. 2).

        Декартово произведение (A × B)

            Возвращает все возможные комбинации кортежей A и B.

            Пример: Все пары "студент — предмет" для формирования учебного плана.

            Степень результирующего отношения = степень A + степень B.

Свойство замкнутости: Результат операции — отношение, что позволяет создавать вложенные запросы.


17. Специальные операции реляционной алгебры

Специальные операции адаптированы для работы с таблицами:

        Выборка (σ / WHERE)

            Фильтрация кортежей по условию.

            Синтаксис: R WHERE условие (например, Студенты WHERE Группа = 382).

            Пример: Сотрудники с зарплатой > 50 000.

        Проекция (π / [ ])

            Выбор подмножества атрибутов.

            Синтаксис: R[Атрибут₁, Атрибут₂] (например, Студенты[ФИО, Группа]).

            Дубликаты кортежей удаляются.

        Соединение (⋈ / JOIN)

            Комбинирует кортежи из двух отношений по совпадающим атрибутам.

            Естественное соединение: Автоматически по общим атрибутам (например, Студенты ⋈ Группы по атрибуту Группа).

            θ-соединение: По произвольному условию (например, A[Цена_прибора > Цена_подставки]B).

        Деление (÷)

            Находит кортежи из R, связанные со всеми кортежами из S.

            Пример: Студенты, сдавшие все экзамены из списка S.

            Синтаксис: R ÷ S, где S — отношение-делитель.

Применение: Операции используются для построения сложных запросов (например, π_ФИО (σ_Группа=382 (Студенты ⋈ Группы))).


 

18. Этапы проектирования базы данных

Проектирование БД — итерационный процесс из трех этапов:

        Концептуальное проектирование

            Анализ предметной области: Выявление сущностей, атрибутов, связей.

            Построение ER-диаграмм (сущности → прямоугольники, связи → ромбы).

            Результат: Инфологическая модель, независимая от СУБД.

        Логическое проектирование

            Преобразование ER-модели в схему БД для конкретной СУБД.

            Правила преобразования:

                Связь 1:M → внешний ключ в таблице-"потомке".

                Связь M:N → новая таблица связи.

            Нормализация отношений (устранение избыточности).

        Физическое проектирование

            Оптимизация хранения данных: Выбор структур файлов, индексов, методов доступа.

            Оценка объема памяти и производительности.

            Результат: Спецификация для реализации БД.

Важно! Этапы могут повторяться для уточнения модели.


 

19. Архитектуры баз данных

Архитектура БД определяет распределение функций между компонентами системы:

        Централизованная

            БД, СУБД и приложения находятся на одном компьютере.

            Недостатки: Низкая масштабируемость, риск отказа.

        Файл-серверная

            Файлы БД хранятся на сервере, обработка — на рабочих станциях.

            Проблемы: Высокая сетевая нагрузка, риск конфликтов.

            Пример: Базы на основе MS Access.

        Клиент-серверная (2-звенная)

            Клиент: Формирует SQL-запросы.

            Сервер БД: Выполняет запросы, возвращает результаты.

            Преимущества: Снижение сетевого трафика, централизованное управление.

        Многоуровневая (3-звенная)

            Клиент: Интерфейс пользователя.

            Сервер приложений: Бизнес-логика.

            Сервер БД: Хранение и обработка данных.

            Пример: Веб-приложения (браузер → веб-сервер → БД).

Тренд: Переход к многоуровневым архитектурам для сложных систем.


20. Модель "Сущность-Связь" (ERD)

ER-модель (модель П. Чена) — инструмент концептуального проектирования БД. Состоит из трех элементов:

        Сущность

            Объект предметной области (студент, отдел).

            Изображается прямоугольником.

            Имеет экземпляры (например, "Иванов А.А.").

        Атрибут

            Свойство сущности (ФИО, номер группы).

            Изображается овалом.

            Ключевые атрибуты подчеркиваются.

        Связь

            Ассоциация между сущностями (студент "учится в" группе).

            Изображается ромбом с указанием типа (1:1, 1:M, M:N).

            Класс принадлежности:

                Обязательный (черный кружок ▉): Все экземпляры участвуют в связи.

                Необязательный (◦): Не все экземпляры участвуют.

Пример ER-диаграммы:

 

[ПРЕПОДАВАТЕЛЬ] ———< ВЕДЕТ >——— [ДИСЦИПЛИНА]

Преобразование ERD в таблицы:

        Связь M:N → новая таблица Ведение с ключами Преподаватель_ID и Дисциплина_ID.

Преимущества: Наглядность, независимость от реализации, основа для нормализации.

21. Диаграммы ER-экземпляров и ER-типа.

Диаграммы ER-экземпляров и ER-типа

ER-модель (Entity-Relationship)

Это графический способ описания структуры данных предметной области. Включает:

·         Сущности – объекты реального мира (например, "Студент", "Группа").

·         Атрибуты – характеристики сущностей ("ФИО", "Номер зачётки").

·         Связи – отношения между сущностями ("Учится в группе").

ER-тип (типов)

·         Это общая схема модели , показывающая типы сущностей, связей и их атрибутов.

·         Не содержит конкретных данных.

·         Пример: "Студент" связан с "Группой" через связь "учится".

ER-экземпляр

·         Это конкретная реализация модели с данными.

·         Показывает экземпляры сущностей и связей.

·         Пример: Студент Иванов учится в группе ИВТ-101.

22.Структурный подход при разработке инфологической модели.

Что такое инфологическая модель?

·         Это концептуальная модель предметной области.

·         Описывает, что хранит система , а не как это делается.

·         Создаётся до проектирования логической или физической структуры базы данных.

Структурный подход предполагает поэтапное проектирование модели данных:

1.      Выделение сущностей (объектов предметной области).

2.      Определение атрибутов и их типов.

3.      Установление связей между сущностями (1:1, 1:M, M:N).

4.      Нормализация для устранения избыточности.

5.      Проверка целостности (первичные/внешние ключи).

Пример: Для университета:

·         Сущности: Студент, Преподаватель, Курс.

·         Связи: Студент посещает Курс, Преподаватель ведет Курс.

Преимущества структурного подхода:

Наглядность.

Возможность согласования с заказчиком.

Простота перехода к логической (реляционной) модели.

23.Целостность сущности и ссылок.

Целостность сущности

·         Каждая строка таблицы должна быть уникально идентифицируема.

·         Реализуется через первичный ключ .

·         Первичный ключ должен быть:

1.      Уникальным

2.      Не пустым (NOT NULL)

Целостность ссылок (ссылочная целостность)

·         Обеспечивает корректность межтабличных связей.

·         Если в одной таблице есть внешний ключ, то он должен:

1.      Либо совпадать со значением первичного ключа в другой таблице,

2.      Либо быть NULL.

Как поддерживается?

·         Средствами СУБД (например, FOREIGN KEY в SQL).

·         При нарушении целостности операции (вставка, удаление) блокируются или выполняются каскадно.

24.Потенциальные, первичные и внешние ключи реляционной модели.

Потенциальный ключ — атрибут или набор атрибутов, уникально идентифицирующий запись (кандидат в ПК).

·         Любой минимальный набор атрибутов, однозначно определяющий запись в таблице.

·         Может быть несколько потенциальных ключей.

Первичный ключ (ПК) — выбранный из потенциальных ключей основной идентификатор.

·         Один из потенциальных ключей, выбранный в качестве основного идентификатора.

·         Обязательно должен быть задан и не может содержать дубликатов или значений NULL.

Пример: ID_студента в таблице Студент.

Внешний ключ (ВК) — атрибут, ссылающийся на ПК другой таблицы.

·         Атрибут или группа атрибутов, ссылающихся на первичный ключ другой таблицы.

·         Используется для установления связей между таблицами.

Пример: ID_курса в таблице Зачетка ссылается на Курс.

25.Понятие нормализации отношений.

Что такое нормализация?

·         Это процесс устранения избыточности и аномалий в структуре базы данных.

·         Заключается в декомпозиции отношений (таблиц) на более простые, соответствующие определённым правилам.

Цели нормализации:

·         Устранить дублирование данных.

·         Предотвратить аномалии при вставке, удалении и обновлении.

·         Улучшить гибкость и согласованность БД.

Основные аномалии:

·         Вставки: Невозможность добавить данные без других данных.

Пример: Нельзя добавить Курс, пока нет Преподавателя.

·         Обновления: Несогласованность при изменении.

Пример: Изменение Цены_товара в одном заказе, но не в других.

·         Удаления: Потеря данных при удалении связанных записей.

Пример: Удаление Студента удаляет все его Зачетки.

26.Первая, вторая и третья нормальные формы.

1НФ (Первая нормальная форма)

·         Все значения атрибутов атомарны (не могут быть разделены).

·         Отсутствуют повторяющиеся группы столбцов.

2НФ (Вторая нормальная форма)

·         Выполняется 1НФ.

·         Все неключевые атрибуты полностью зависят от всего первичного ключа (нет частичных зависимостей)

3НФ (Третья нормальная форма)

·         Выполняется 2НФ.

·         Нет транзитивных зависимостей неключевых атрибутов от первичного ключа (все неключевые атрибуты зависят только от первичного ключа)

27.Нормальная форма Бойса-Кодда, четвертая и пятая нормальные формы.

НФБК (нормальная форма Бойса-Кодда)

·         Усиленная версия 3НФ.

·         Для любой функциональной зависимости X → Y, X должно быть супер ключом .

Пример: Если A → B и A не является ключом → нарушение НФБК.

Зачем нужна НФБК?

·         Чтобы исключить случаи, когда 3НФ выполнена, но аномалии всё ещё возможны.

4НФ (Четвертая нормальная форма)

·         Устраняет многозначные зависимости .

·         Отношение находится в 4НФ, если оно находится в НФБК и любая многозначная зависимость тривиальна.

Многозначная зависимость:
A многозначно определяет B, если для данного A множество B определяется независимо от C.

 

5НФ (Пятая нормальная форма / Проекционно-соединительная форма)

·         Исключает зависимости соединения .

·         Отношение находится в 5НФ, если любую его декомпозицию можно восстановить без потери данных.

Пример: Если таблица содержит информацию о поставщиках, деталях и проектах и при этом эти данные связаны сложными условиями, то её можно разбить на три таблицы.

39. Типы данных SQL.

  1. CHAR(n): Строка фиксированной длины.

42. Организация запросов на выборку данных при помощи языка SQL.

Основным оператором для извлечения данных в SQL является SELECT. Этот оператор позволяет не только выполнять простую выборку строк из таблицы (горизонтальное подмножество), но и осуществлять соединение нескольких таблиц.

Полный синтаксис оператора SELECT выглядит следующим образом:
SELECT [ALL | DISTINCT] <список_выбора>
FROM <имя_таблицы>
[WHERE <условие>]
[GROUP BY <имя_столбца>, …]
[HAVING <условие>]
[ORDER BY <имя_столбца> [ASC | DESC], …]

При написании запроса важно соблюдать строгий порядок предложений - например, GROUP BY всегда должно предшествовать ORDER BY. В списке выбора можно указать конкретные столбцы через запятую или использовать символ * для выбора всех столбцов.

Ключевое слово DISTINCT позволяет исключить дублирующиеся строки из результата, в то время как ALL (используемое по умолчанию) оставляет все строки. Условие WHERE служит для фильтрации записей по заданным критериям.

При выборке из нескольких таблиц их имена перечисляются в предложении FROM, а условие соединения указывается в WHERE. Если в разных таблицах есть столбцы с одинаковыми именами, необходимо использовать полную квалификацию вида <таблица>.<столбец>. Оператор SELECT поддерживает соединение не только двух, но и большего количества таблиц.

43. Сортировка и группировка данных при помощи языка SQL.

Группировка данных в операторе SELECT осуществляется с помощью ключевого слова GROUP BY и ключевого слова HAVING, с помощью которого задаются условия разбиения записей на группы.

GROUP BY неразрывно связано с агрегирующими функциями, без них оно практически не используется. GROUP BY разделяет таблицу на группы, а агрегирующая функция вычисляет для каждой из них итоговое значение.

Ключевое слово HAVING работает следующим образом: сначала GROUP BY разбивает строки на группы, затем на полученные наборы накладываются условия HAVING.

Для сортировки данных, получаемых при помощи оператора SELECT, служит ключевое слово ORDER BY. С его помощью можно сортировать результаты по любому столбцу или выражению, указанному в <список_выбора>. Данные могут быть упорядочены как по возрастанию, так и по убыванию. Ключевое слово DESC задает обратный порядок сортировки, а ключевое слово ASC (его можно опустить) — прямой порядок сортировки.

44. Локальные и сетевые базы данных.

Локальная база данных располагается на том же компьютере, что и использующее её приложение, обеспечивая доступ к данным только с этого устройства. Она может выступать как вспомогательный набор строк, подключаемый к основной базе, что полезно при необходимости хранения персональных данных пользователей в рамках общей системы.

Сетевая база данных организована по принципу множественного владения записями, предоставляя различные пути доступа к информации. Такие системы известны как СУБД CODASYL (Conference on Data Systems Languages).

Централизованная база данных полностью размещается на одной вычислительной системе - будь то пользовательский компьютер или сервер. Доступ осуществляется через специальный драйвер или напрямую. При подключении системы к сети возможен удалённый доступ с других устройств, хотя локальные базы без сетевого доступа не поддерживают одновременную работу нескольких пользователей.

В файл-серверных системах с сетевым доступом выделяется центральный сервер для хранения общей базы данных. Остальные компьютеры функционируют как рабочие станции, получающие файлы базы для обработки. Такая архитектура может снижать производительность при интенсивном обращении к одним и тем же данным.

Клиент-серверная архитектура предполагает, что сервер базы данных не только хранит информацию, но и выполняет основную обработку. Рабочие станции формируют запросы, которые обрабатываются на сервере, после чего результаты передаются обратно клиенту.

45. Распределенные БД.

Распределенная база данных (РБД) размещается на нескольких компьютерах (обычно серверах), соединенных сетью. Каждый узел такой системы представляет собой полноценную СУБД, но взаимодействие между узлами организовано так, что пользователь получает доступ ко всем данным сети, как если бы они хранились локально. Фактически РБД — это виртуальная база, объединяющая физически разрозненные данные.

Ключевые особенности:

  • Каждый узел имеет свои локальные базы, пользователей, СУБД и механизмы управления транзакциями.
  • Система обеспечивает:
  • Простоту использования (доступ к данным как к единому целому).
  • Автономность работы узлов при сбоях сети.
  • Высокую эффективность обработки запросов.

Типы распределенных систем:

  1. Однородные — все узлы используют одинаковые СУБД.
  2. Неоднородные — узлы могут работать с разными моделями данных (интеграция таких систем сложна и часто неэффективна).

Пример: System R*
Цель — интеграция локальных баз System R в сети с сохранением прозрачности доступа. Основные решения:

  • Прозрачность местоположения данных — система автоматически определяет, где хранятся запрашиваемые данные.
  • Автономность узлов — каждый узел администрируется независимо, работа продолжается даже при сбоях сети.
  • Оптимизация запросов — компиляция запросов с выбором оптимального плана выполнения, включая распределенную обработку.
  • Поддержка SQL — пользователи работают в привычной среде без необходимости учитывать распределенность данных.

Не реализованные возможности в System R*:

  • Горизонтальное и вертикальное разделение отношений.
  • Дублирование данных с автоматической синхронизацией копий.
  • Мгновенные снимки (снэпшоты) данных для периодического обновления.

Проблемы:

  • Сложность поддержания согласованности разделенных или дублированных данных.
  • Низкая эффективность при обработке запросов к распределенным отношениям.
  • Высокие накладные расходы при синхронизации и восстановлении после сбоев.

Итог:
Распределенные БД актуальны для организаций с территориально разнесенными подразделениями. Однако их внедрение требует решения сложных задач по оптимизации, синхронизации и отказоустойчивости. System R* стала важным шагом в этом направлении, но многие теоретические возможности пока не реализованы на практике.

46. Современные реляционные СУБД.

Системы управления базами данных (СУБД) играют ключевую роль в хранении и обработке информации. Они постоянно совершенствуются, поддерживая сложные структуры данных и интегрируясь с сетевыми и телекоммуникационными технологиями.

Реляционные СУБД (RDBMS) представляют данные в виде таблиц. Основной язык взаимодействия с ними — SQL (Structured Query Language), который стал стандартом в отрасли. Он позволяет выполнять выборку, обновление и управление данными. Важные понятия: первичный ключ (уникально идентифицирует запись), внешний ключ (связывает таблицы). Недостатки реляционных БД — сложность при создании гибких связей и высокая трудоёмкость при изменении структуры.

1.1 Microsoft Access

MS Access — простая и популярная СУБД от Microsoft, входящая в пакет Office. Подходит для небольших проектов. Имеет ограничения: низкая производительность из-за постоянного сохранения, отсутствие компилятора EXE, зависимость от Windows.

1.2 Microsoft SQL Server

Мощная серверная СУБД от Microsoft. Использует язык Transact-SQL (T-SQL), расширяющий SQL. Поддерживает большие объемы данных, кластеризацию, зеркалирование, репликацию. Интегрирована с .NET Framework, имеет широкие возможности для разработки и масштабирования. Поддерживает подключение через ODBC, JDBC, SOAP и веб-сервисы.

1.3 MySQL

Бесплатная и популярная СУБД, распространяемая Oracle. Подходит для малых и средних приложений, часто используется в веб-разработке (входит в LAMP/WAMP). Поддерживает разные типы таблиц (MyISAM, InnoDB), транзакции, полнотекстовый поиск. Имеет API для множества языков программирования.

1.4 IBM DB2

Высокопроизводительная СУБД от IBM. Отличается мощным оптимизатором запросов, строгой декларативностью SQL и глубокой интеграцией с корпоративными системами безопасности. Поддерживает XML, рекурсивные запросы, табличные выражения. Используется в крупных предприятиях, особенно на мейнфреймах.

47. Современные объектно-ориентированные (объектные) СУБД.

Современные системы управления базами данных обеспечивают высочайшую производительность, неограниченную масштабируемость, надежность и отказоустойчивость. Эти характеристики особенно важны для корпоративных решений, где требуется обработка больших объемов информации в режиме реального времени.

Объектно-ориентированные СУБД
Объектно-ориентированные системы управления базами данных (ООСУБД) представляют собой новый подход к организации данных, где информация обрабатывается как абстрактные объекты, обладающие свойствами и методами. Такие системы появились в середине 1980-х годов как ответ на потребности сложных информационных систем, где традиционные реляционные модели оказывались недостаточно гибкими.

Основные концепции ООСУБД включают:

  • Объекты с уникальными идентификаторами
  • Атрибуты и методы
  • Классы и их иерархию
  • Наследование свойств и поведений

СУБД Cache
Cache представляет собой передовую пост-реляционную объектную СУБД, которая сочетает в себе преимущества различных подходов к работе с данными. Эта система особенно эффективна для:

  • Разработки веб-решений
  • Создания клиент-серверных приложений
  • Обработки транзакций в системах с большими объемами данных

Ключевые особенности Cache:

  1. Многомодельный доступ к данным:
  • Объектный доступ с полной поддержкой ООП-принципов
  • Реляционный SQL-доступ через JDBC/ODBC
  • Прямой доступ к многомерным данным
  1. Производительность и масштабируемость:
  • Обработка десятков тысяч транзакций
  • Поддержка кластеризации и зеркалирования
  • Возможность анализа данных в реальном времени
  1. Гибкость разработки:
  • Поддержка различных языков программирования (Caché ObjectScript, Caché Basic)
  • Интеграция с .NET, Java, COM и C++
  • Совместимость с реляционными базами данных через SQL-Gateway

Преимущества использования Cache:

  • Быстрое моделирование сложных структур данных
  • Высокая производительность критически важных участков кода
  • Возможность использования привычного SQL-инструментария
  • Поддержка унаследованных приложений
  • Развитая среда проектирования и разработки

Cache представляет собой универсальное решение для предприятий, которым требуется надежная, производительная и гибкая система управления данными. Его уникальная архитектура позволяет эффективно решать самые сложные задачи обработки информации, сочетая преимущества объектного, реляционного и многомерного подходов.

48. Объектно-реляционная СУБД (ОРСУБД)

Объектно-реляционная СУБД (ОРСУБД) представляет собой гибридную систему, которая сочетает проверенные временем реляционные технологии с перспективными объектно-ориентированными подходами. Такие системы сохраняют все преимущества реляционных баз данных, одновременно предлагая возможности работы с объектами.

Преимущества реляционного подхода
Несмотря на появление новых технологий, реляционные СУБД сохраняют свою актуальность благодаря ряду неоспоримых достоинств:

  • Возможность раздельного использования таблиц различными приложениями
  • Подробная система диагностики ошибок с развернутыми кодами возврата
  • Высокая производительность при обработке запросов, особенно при использовании оператора SELECT
  • Относительно простая концепция, не требующая длительного обучения
  • Эффективная работа с большими объемами информации

Гибридные решения
В современных условиях многие организации предпочитают постепенный переход к новым технологиям, используя следующие подходы:

  1. Объектно-реляционные адаптеры
  • Автоматически преобразуют объекты в реляционные структуры
  • Позволяют сохранять инвестиции в существующие реляционные системы
  • Примеры: Odapter от Hewlett-Packard, функциональность GemStone
  1. Объектно-реляционные шлюзы
  • Транслируют объектно-ориентированные запросы в реляционные
  • Обеспечивают работу с уникальными идентификаторами объектов (OID)
  • Могут работать на разных уровнях абстракции СУБД
  1. Гибридные СУБД
  • Нативно поддерживают обе модели данных
  • Позволяют постепенно внедрять объектные технологии
  • Представляют собой наиболее перспективное направление развития

Oracle Database как пример успешной ОРСУБД
Компания Oracle уже более десяти лет демонстрирует успешную реализацию объектно-реляционного подхода в своих продуктах. Современные версии Oracle Database предлагают:

  • Гибкую архитектуру с единой кодовой базой для различных редакций
  • Надежную систему безопасности, включая:
  • Детализированный контроль доступа
  • Встроенные механизмы аудита
  • Средства резервного копирования и восстановления
  • Защиту от физических и логических повреждений
  • Масштабируемость от персональных решений до корпоративных систем
  • Высокую производительность благодаря оптимизированному ядру

Заключение
Объектно-реляционные СУБД представляют собой разумный компромисс между проверенными реляционными технологиями и перспективными объектными подходами. Такие системы, как Oracle Database, демонстрируют, что гибридный подход позволяет организациям постепенно внедрять новые технологии, сохраняя инвестиции в существующую инфраструктуру и обеспечивая плавный переход к более современным методам работы с данными.
